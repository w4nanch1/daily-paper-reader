# Environment-free Synthetic Data Generation for API-Calling Agents 

Seanie Lee , Sanjoy Chowdhury , Chao Jiang , Cheng-Yu Hsieh , Ting-Yao Hu , Alexander Toshev , Oncel Tuzel ,

Raviteja Vemulapalli 

Apple Training API-calling large language model (LLM) agents demands massive amounts of high-quality trajectories. How-ever, collecting such data at scale typically requires fully implemented environments with executable APIs and realis-tic, pre-populated backend databases, creating a major bottleneck for scalability. To overcome this, we propose an environment-free synthetic data generation approach that leverages LLMs as on-the-fly digital world models. Given only API specifications, our method generates trajectories mimicking interactions between an agent and a stateful environ-ment. Specifically, an LLM first generates diverse tasks solvable with the provided APIs. A teacher agent then iteratively solves each task while an LLM simulator generates coherent synthetic API responses conditioned on the task context and simulation history. Finally, an LLM judge filters the trajectories to ensure the quality of the resulting dataset. We evaluate our approach on the challenging AppWorld and OfficeBench benchmarks, which include both information-retrieval and state-changing tasks. Fine-tuning models on our synthetic data yields significant performance gains, demonstrating that effective supervision for API-calling agents can be generated without any executable environment. Our results establish LLM-based API simulation as a practical, scalable solution for training agents across diverse API ecosystems.

## 1 Introduction 

The rapid advancement of large language models (LLMs) has driven a paradigm shift from passive text generators to active, API-calling agents (Schick et al., 2023; Yao et al., 2023). These agents fulfill complex user requests by executing API calls and reasoning over environmental feedback. However, training such agents requires vast amounts of high-quality trajectory data that capture the full multi-step interaction process, including user instructions, sequences of interleaved read and write API calls, execution results, and final responses. Compared to standard instruction-tuning data, these multi-step trajectories are substantially harder to collect at scale, yet they remain essential for teaching agents robust API usage, accurate state tracking, and effective error recovery. Given the difficulty of manual collection, synthetic data generation has emerged as a natural solution, with recent works leveraging LLMs to produce agentic trajectories at scale (Qin et al., 2024; Zeng et al., 2024; Liu et al., 2024; Xu et al., 2026; Lu et al., 2026). However, these works assume access to fully implemented environments with executable APIs and backend databases filled with realistic underlying states. This tight coupling between environment creation and data synthesis creates a major scalability bottleneck. To generate training data for a new set of APIs, one must first engineer the corresponding infrastructure and, more challengingly, source the diverse underlying state configurations required to populate the environment. In this work, we ask a fundamental question: can we generate high-quality agentic trajectory data without building the underlying environments at all ? Our key conjecture is that modern LLMs, having been pretrained on vast corpora of world knowledge, can function as dynamic digital world models. Driven by this insight, we introduce a highly scalable paradigm for agentic data generation. Instead of constructing real environments and executing actual APIs, we use LLMs to simulate all environmental feedback on-the-fly directly from API specifications. To instantiate this paradigm, we propose ESAT , an Environment-Free Synthetic Agentic Trajectory gener-ation pipeline that uses only API specifications as input. The pipeline operates in three stages, as illustrated in Fig. 1. First, during the task synthesis stage, an LLM generates diverse, realistic tasks solvable by the target APIs, with an LLM-based critic filtering out invalid tasks. Second, the core trajectory synthesis stage pairs a teacher agent with our LLM-based API simulator. As the teacher iteratively issues API calls to solve the task, the simulator seamlessly generates coherent environmental feedback on-the-fly using the user task and prior simulation history as context. This interaction produces complete, multi-step agent trajectories entirely in simulation, without ever requiring an executable environment (see Sec. L for example trajecto-ries). Finally, during trajectory filtering , an LLM judge evaluates the generated trajectories for correctness, completeness, and redundancy, discarding sub-optimal samples to ensure a high-fidelity training dataset. 

ESAT offers two critical advantages. First, it turns agentic trajectory synthesis into a lightweight speci-fication problem. By relying solely on API specifications rather than fully implemented environments or real-world deployed services, it allows data generation to scale seamlessly across diverse API ecosystems. Second, leveraging the long-context capabilities of modern LLMs, our simulator maintains state coherence across successive API calls, allowing for the generation of rich, multi-turn training trajectories that go far beyond isolated API-calling examples. To validate ESAT , we conduct extensive experiments on AppWorld (Trivedi et al., 2024) and OfficeBench (Wang et al., 2024), two challenging multi-app benchmarks that evaluate agents on complex, multi-step information-retrieval and state-changing tasks. Our experiments across models ranging from 1.7B to 27B parameters demonstrate that supervised fine-tuning (SFT) on ESAT data leads to significant performance improvements with respect to the corresponding base models, yielding up to 50.5% on AppWorld and 60.5% on OfficeBench. Most notably, on AppWorld, which provides an official set of training tasks, models trained on ESAT data outperform the models trained on the training task trajectories collected from the real AppWorld environment. Our contributions are summarized below: 

• We introduce a context-aware, LLM-based API simulator that acts as a dynamic digital world model. By leveraging task context and interaction history, this simulator generates coherent environmental feedback, enabling the synthesis of complex, multi-step, API-calling trajectories without any real API execution. 

• We propose ESAT , a novel, environment-free data synthesis pipeline that reduces agentic trajectory gen-eration to a lightweight specification problem. By requiring only API specifications as input, ESAT 

eliminates the infrastructure and data bottlenecks associated with building and populating executable environments, enabling seamless scaling across diverse API ecosystems. 

• We demonstrate that ESAT data is highly effective for supervised fine-tuning. Across models ranging from 1.7B to 27B parameters, our synthetic data drives substantial gains on AppWorld and OfficeBench datasets. 22 Related Work  

> Synthetic data with functional environments.

Because gathering human-annotated interaction data is notori-ously resource-intensive, recent research trends have shifted toward LLM-driven generation. A variety of contemporary works now utilize LLMs to automatically synthesize agentic trajectories at scale (Qin et al., 2024; Zeng et al., 2024; Liu et al., 2024; Xu et al., 2026; Lu et al., 2026). To ensure data fidelity, these approaches execute generated tool calls against live API endpoints or fully implemented sandboxes. While effective for generating grounded trajectories, these methods are fundamentally constrained by their depen-dence on functional environments. Adapting these pipelines to new APIs requires building bespoke backend infrastructures and painstakingly sourcing the diverse state data required to populate them. Consequently, this overhead severely limits the speed and breadth with which these methods can scale to new API domains.  

> LLM as an API simulator.

To bypass the bottleneck of executable backends, several existing works employ LLMs to simulate API responses. Early works such as ToolAlpaca (Tang et al., 2023) and ToolACE (Liu et al., 2025), and recent efforts like Nemotron (Chandiramani et al., 2026), synthesize multi-turn data by prompting LLMs to generate mock execution results. However, these methods rely on the LLM’s implicit conversational coherence, which leaves them vulnerable to state drift and hallucinated identifiers in long, complex trajectories. Other approaches tailor simulation to specific use cases: ToolEmu (Ruan et al., 2024) emulates outputs for safety evaluation, while StableToolBench (Guo et al., 2024) uses a GPT-4 simulator as a fallback for the offline evaluation of read APIs. The closest work to ours is Simia (Li et al., 2025), which utilizes simulated environments for agent training by augmenting seed trajectories for SFT and replacing real environments with simulated feedback during RL. 

ESAT differs from prior simulation pipelines in four key ways. First, while most simulators focus on read APIs (Tang et al., 2023; Guo et al., 2024), ESAT supports state-changing write APIs, enabling interactions within dynamic, stateful environments. Second, unlike methods requiring pre-compiled API pools (Liu et al., 2025), existing benchmarks (Chandiramani et al., 2026), or seed trajectories (Li et al., 2025), ESAT generates both tasks and trajectories from scratch using only API specifications, making it seamlessly adaptable to new API domains. Third, instead of generating a full trajectory in a single LLM pass as in Li et al. (2025), 

ESAT uses explicit, step-by-step interactions between separate agent and simulator LLMs to generate multi-step trajectories. Fourth, to ensure state consistency across these long trajectories, the ESAT simulator validates each simulated response against the full interaction history, thereby acting as a persistent digital world model. 

## 3 Method 

In this section, we present ESAT , our environment-free pipeline that transforms API specifications into a dataset of multi-step, API-calling trajectories without relying on real API execution, backend infrastructure, or human-written tasks. As illustrated in Fig. 1, the ESAT pipeline consists of three stages: (i) task synthesis from API specifications, (ii) trajectory synthesis using a teacher agent paired with an LLM-based API simulator, and (iii) trajectory filtering via an LLM judge. 

3.1 Task Synthesis 

This stage generates diverse user tasks solvable using the available APIs. The sole required input is a set of API specifications detailing each API’s name, purpose, and input/output schemas.  

> Bucketized generation.

To ensure broad and balanced coverage, task generation is organized into a combina-torial grid of configuration buckets . Each bucket is defined by a tuple of attributes, including difficulty level 

(e.g. , easy, medium, hard), action type (e.g. , read, write, mixed), task focus (e.g. , constraint satisfaction, derivation), the number of apps (1, 2, or 3), and a range of APIs per app . These buckets are processed in a randomized sequence, and a task generator LLM is prompted multiple times to yield a desired number of tasks per bucket. During each invocation, we randomly sample apps matching the bucket’s required number and provide their full API specifications to the LLM, alongside the target difficulty, action type, task focus, 3and API count. The LLM then generates a task along with the list of APIs required to solve it. Detailed descriptions of the configuration attributes and their values are provided in Sec. G.1.  

> Inverse frequency-based sampling.

To ensure uniform coverage of apps and APIs, we employ an inverse frequency-based sampling strategy. We maintain running usage counts for every app and API across all generated tasks. When selecting apps for a generation step, we sample them with probabilities inversely proportional to their current counts, thereby prioritizing underrepresented apps. Furthermore, we rank all available APIs within the selected apps based on their counts, and explicitly instruct the task generator LLM to focus on the least-used APIs.  

> Task filtering.

Each generated task is evaluated by an LLM judge to ensure it is well-formed, solvable, and consistent with the corresponding bucket configuration.  

> Task rewriting.

Tasks generated directly from API specifications can sometimes suffer from procedural ver-bosity, outlining detailed instructions rather than natural goals. To address this, all tasks accepted by the judge are subsequently rewritten by an LLM to compress any procedural scaffolding into concise, intent-level requests that reflect actual user needs. These rewritten tasks are then verified by an LLM judge to ensure they remain valid. Sec. G.4 provides examples illustrating the impact of this rewriting process. If a rewritten task is rejected by the judge, the corresponding initial task is retained for trajectory synthesis. 

3.2 Trajectory Synthesis 

In this stage, a teacher LLM agent iteratively solves each task by issuing API calls, while an LLM-based API simulator dynamically generates synthetic responses. This yields complete, multi-step training trajectories that include the agent’s reasoning, API calls, and simulated environmental observations, all without executing any real APIs. Example trajectories are provided in Sec. L. To ensure structural correctness and semantic coherence, the simulator LLM conditions its output on a rich input context, which includes: (1) the full API specification, (2) the API call arguments, (3) the task being solved, (4) the history of previous API calls in the trajectory and their simulated responses, (5) the name, email, and phone number of a virtual user, and (6) the current date and time. To prevent overly long input contexts and accelerate simulation, we restrict the simulation history to previous API calls from the same app.  

> Rules for the simulator.

The simulator LLM is guided by four core rules encoded in its system prompt: (1) 

Data consistency : it must preserve state across turns ( e.g. , a song assigned to a specific artist earlier must remain under that artist); (2) Data diversity : listed results should contain a mix of task-relevant and irrelevant items to mirror real-world noise; (3) Data realism : generated values must respect field constraints such as numeric ranges and unique identifiers; and (4) Schema compliance : responses must strictly conform to the schema provided in the API specification.  

> Simulation quality checks.

To ensure that the simulator produces accurate and realistic responses, we employ multiple stages of quality control. Before invoking the simulator LLM, the API call arguments are program-matically validated for missing parameters and type mismatches. If errors are detected, the simulator directly returns an appropriate error message without invoking the LLM. For valid API calls, the LLM-generated re-sponse undergoes multiple checks. First, it is programmatically verified against the expected output schema. In the event of a schema mismatch, the simulator LLM is re-prompted using the schema errors as feedback. Once the response passes the schema check, it is evaluated for semantic correctness and consistency with the simulation history using the same LLM as a judge. If the judge identifies any errors, the simulator LLM is prompted to refine its response using the judge’s output as feedback. Only responses that pass all checks are returned. We allow a fixed number of refinement steps; if the simulator fails to generate a valid response after exhausting all retries, we declare a simulation failure. 4Trajectory termination. Trajectory generation for a task concludes when the agent explicitly calls a special task completion API, exceeds a predefined limit on the number of execution steps, or invokes an API from an app that was not used to synthesize the corresponding task. The process also halts if the simulator fails. 

3.3 Trajectory Filtering 

Trajectories in which the agent finished the task by calling the task completion API are finally evaluated using a trajectory-level LLM judge, which decides whether the agent correctly solved the corresponding task. Each trajectory is judged multiple times, and we retain only those with a majority-positive verdict. All the LLM prompts used for data synthesis are provided in Sec. H.7. 

## 4 Experiments  

> Benchmarks.

We evaluated our approach on two challenging, multi-app benchmarks designed to test complex API interactions. AppWorld (Trivedi et al., 2024) provides a rigorous testbed of intricate tasks that require agents to coordinate numerous read and write API calls across multiple apps. The environment features 457 APIs distributed across nine everyday apps, with the average task necessitating 10 unique APIs and 45 total calls. AppWorld provides two test splits: Test-Normal (Test-N) with 168 tasks and Test-Challenge (Test-C) with 417 tasks. The latter introduces two held-out apps to evaluate generalization under increased difficulty. OfficeBench (Wang et al., 2024) focuses on office automation, featuring 20 APIs across eight apps. Tasks require agents to plan long-horizon workflows across apps, such as extracting spreadsheet data, scheduling calendar events, and dispatching emails. Following Li et al. (2025), we excluded OCR-dependent and unsolvable tasks, evaluating on 26 two-app and 46 three-app tasks.  

> Models.

We used GLM-4.7-FP8 (Zeng et al., 2025) for task synthesis, GLM-5.1-FP8 (Zeng et al., 2026) for trajectory synthesis (the teacher agent and API simulator), and Gemini-3.1-Pro (Google DeepMind, 2026) for trajectory filtering. The resulting synthetic data was used to fine-tune eight models: Qwen3 (1.7B, 4B, 8B, and 14B) (Yang et al., 2025) and Qwen3.5 (2B, 4B, 9B, and 27B) (Qwen Team, 2026).  

> Training.

Each training trajectory was formatted as a multi-turn conversation consisting of the task instruc-tion, the agent’s reasoning and API calls, and the simulated API responses. We followed a standard SFT strategy, training the model to predict the agent’s output in each turn conditioned on the task instruction and all previous API calls and their responses. Detailed training hyperparameters are provided in Sec. D.  

> Evaluation.

All results reported in this section are averaged over eight inference runs, with standard devia-tions presented in Sec. F.

4.3 LLM Judge Quality 

To understand the effect of the judge model used for trajectory filtering, we performed an ablation study comparing GLM-5.1-FP8 and Gemini-3.1-Pro. Table 3 shows the performance of the Qwen3-8B model trained on ESAT-S52-AW7 data filtered by each of these models. Compared to an unfiltered baseline, employing GLM for trajectory curation actually degrades the performance. In contrast, Gemini proves to be a highly effective filter. By aggressively discarding a larger proportion of low-quality trajectories, it ultimately yields superior downstream performance.  

> Correlation with real-environment correctness.

To quantify how well our LLM judge corresponds to true ex-ecutable correctness, we conducted an empirical validation against the real AppWorld environment. We generated 720 trajectories (8 per task across the 90 AppWorld training tasks) using GLM-5.1-FP8 as the agent, and obtained ground-truth trajectory labels from AppWorld’s execution-based verifiers. We then re-evaluated these trajectories using our LLM judge, Gemini-3.1-Pro. The judge achieved a precision of 95.2%, confirming that when our pipeline flags a trajectory as correct, it is highly likely to be genuinely correct. This is the property that matters most for data curation, as it bounds the proportion of erroneous trajectories admitted into the training set.  

> Agreement with human annotators.

To further validate the trajectory filtering stage, we conducted a human evaluation of our LLM judge. > Effect of response length.

Fig. 2 shows how the simulation failure rate varies with the length of the simulated response. For short responses under 500 tokens, the failure rate remains low (2–3%), indicating that the simulator reliably produces well-formed responses when the required output is compact. The failure rate then rises steadily with length, reaching 8% in the 500–1K range, 18% for 1K–2K tokens, and 23% beyond 2K tokens. 4.5 Comparison with Existing Synthetic API-calling Datasets 

We compare the ESAT dataset against three existing synthetic API-calling datasets. The ToolAlpaca (Tang et al., 2023) and ToolACE (Liu et al., 2025) datasets target simple API-calling scenarios and do not model a stateful environment; specifically, their instances do not require coordinated read/write sequences over an evolving environment state. Nemotron (Chandiramani et al., 2026) is a recent work that scales synthetic API-calling data to 1.5M trajectories, but its API simulator generates each response at the turn level against a correctness rubric rather than a maintained state. Furthermore, its single-and multi-turn scenarios are not designed for the long-horizon, multi-app workflows present in AppWorld and OfficeBench. In contrast, 

ESAT ’s simulator acts as a stateful digital world model that maintains a consistent virtual state across all API calls within a task. Table 4 and Table 5 compare the effectiveness of these synthetic API-calling datasets in terms of downstream performance on the AppWorld and OfficeBench datasets, respectively. Finetuning models on the ToolAlpaca and ToolACE datasets yields performance that is similar to or worse than the corresponding zero-shot accuracy, while finetuning on the Nemotron dataset leads to only marginal performance gains in some cases. This clearly demonstrates that these simple tool-calling datasets do not equip agents with the skills needed to navigate complex, stateful environments. In contrast, training on the ESAT dataset leads to 10 substantial performance gains on both benchmarks. To test if these alternative synthetic datasets offer useful foundational skills, we further fine-tuned the models on the AWT data collected from the AppWorld environment. Even with this real-environment supervision, their final performance was significantly lower than that of models trained on ESAT data. 

## 5 Conclusion 

We presented ESAT , an environment-free pipeline that synthesizes training data for API-calling agents using only API specifications. By combining coverage-aware task generation, a state-consistent LLM-based API simulator, and a trajectory-level LLM judge, ESAT produces high-quality, multi-step trajectories without using any functional environment. Evaluations on the challenging AppWorld and OfficeBench benchmarks demonstrate substantial performance gains across models ranging from 1.7B to 27B parameters. On Ap-pWorld, models trained on synthetic ESAT data outperform the models trained on trajectories collected from the real environment. On both benchmarks, we are able to either outperform or approach the perfor-mance of GPT-4o using significantly smaller models with 4B to 8B parameters. These results establish our environment-free data synthesis pipeline as a highly effective and scalable solution for training API-calling agents.

> Training.

We fine-tune all models with supervised learning on full multi-turn trajectories. Each trajec-tory is decomposed into individual turns, and the model is trained to predict the tokens of the current turn conditioned on all preceding turns, with the loss masked out over the context tokens. We use the AdamW (Loshchilov and Hutter, 2019) optimizer with a learning rate of 2 · 10 −5, a linear decay schedule, and a warmup ratio of 1%. Training is performed with a context length of 32,768 tokens and a per-device batch size of 1. For models smaller than 14B parameters, training is distributed across 8 H100 GPUs, whereas for models 14B and larger, we utilize 8 B200 GPUs. We leverage the Transformers library (Wolf et al., 2020) for model implementation and use PyTorch native FSDP2 for distributed training. To improve training efficiency, we apply sequence packing, which concatenates roughly 4 samples into a single sequence on average. We train for 10 epochs by default, except for the Qwen3.5 family, where we observed overfitting beyond 5 epochs and therefore reduce training to 5 epochs. For the first stage of training on synthetic data, we use both the AppWorld train and dev splits for checkpoint selection. For the second stage, which further fine-tunes the model on the AppWorld train split, we keep all hyperparameters identical and select the best checkpoint based on the dev split alone. For OfficeBench, as there is no dedicated development set, we apply the exact same hyperparameters used for AppWorld.  

> Inference and evaluation.

For efficient, high-throughput inference, we evaluate all models using vLLM (Kwon et al., 2023). During evaluation, we set the sampling temperature to 1.0. Following the evaluation protocol of AppWorld and OfficeBench, the maximum number of agent steps is capped at 50 for AppWorld and 30 for OfficeBench. For both benchmarks, we generate 8 independent samples per task and report the average pass@1 metric to robustly measure agent performance.

## G Task Synthesis 

The goal of task synthesis is to produce a large and diverse pool of natural, multi-step user tasks that can be solved using only the available APIs. The only input to this stage is a set of API specifications, where each specification provides an API’s name, purpose, input-parameter schema, and output-response schema. From these specifications, we generate, judge, and rewrite candidate tasks. We use a single LLM, GLM-4.7-FP8 (Zeng et al., 2025), for all three roles (generator, judge, and rewriter) by varying the prompt. 24 G.1 Bucketized generation. 

To ensure broad and balanced coverage of the task space, generation is organized into a combinatorial grid of configuration buckets . Each bucket is a tuple (difficulty, action_type, task_focus, num_apps, [min_apis, max_apis]), where (i) difficulty ∈ { easy , medium , hard } controls the depth of reasoning required (Table 19); (ii) action_type ∈ { read , write , mixed } controls the mix of GET vs. POST /PATCH /DELETE calls (Table 20); (iii) task_focus (e.g. , constraint satisfaction, derivation, iteration, open) controls the high-level reasoning pattern (Table 21); (iv) num_apps ∈ { 1, 2, 3} specifies the number of distinct apps a task must span; and (v) [min_apis , max_apis ] specifies the per-app API budget. The per-app API budgets are pinned to match the typical per-app load needed at each app count: for num_apps = 1 , we sweep five pinned values [k, k ]

for k ∈ { 2, . . . , 6}; for num_apps = 2 , we use ranges {[1 , 2] , [2 , 3] , [3 , 4] }; and for num_apps = 3 , we use 

{[1 , 2] , [2 , 3] }. This yields 3 · 3 · 4 · (5 + 3 + 2) = 360 buckets in total. Buckets are processed sequentially in a random order, and each bucket targets a fixed number of accepted tasks, Nbucket .For each bucket, the pipeline issues ⌈Nbucket /n per_call ⌉ parallel calls to the task generator LLM, each asked to produce nper_call distinct candidate tasks. For hard buckets (difficulty = hard , num_apps = 3 , or num_apps · min_apis > 5), we double the maximum-completion-tokens budget for the generator, judge, and rewriter to accommodate longer outputs. 

G.2 Inverse frequency-based sampling. 

To prevent the generator from collapsing onto a small set of apps and APIs, we maintain running usage counts c(a) and c(α) for every app a and every API α across all tasks generated so far. Coverage is refreshed at the start of each bucket from this job’s own outputs. When constructing a generator call: 1. App sampling. ## H Simulator Details 

This section provides a detailed description of the LLM-based API simulator used during trajectory synthesis. The simulator is purely text-based: there is no real backend, no database, and no network call. Its job is to take an API call issued by the teacher agent and produce a JSON response that (i) conforms to the documented response schema, (ii) is internally consistent with all responses returned so far in the same task, and (iii) plausibly reflects the user profile and the current task. 27 Focus Clause 

constraint satisfaction The task should require the agent to find or select entities that satisfy MULTIPLE simultaneous criteria. The agent must filter, search, or verify that several conditions hold at the same time before acting. Examples of constraints: price within a range, rating above a threshold, date before a deadline, belonging to a specific category, matching a text pattern, being associated with a particular person. The task should require at least 2–3 distinct constraints to be checked together. The challenge is in correctly combining the constraints, not in the action itself. 

derivation The task should require the agent to derive a SINGLE piece of new information by aggregating or comparing multiple data points, then act on that derived result. The derived value is essential — the agent cannot complete the task without first computing it. Examples: compute total spending to decide a budget, find who owes the most and message that one person, calculate average rating to set a threshold, determine which playlist has the longest total duration and play it. The task must involve an aggregation step (sum, average, max, min, count, ranking, or comparison) whose SINGLE result drives a subsequent action. The final action should target one entity or a fixed number of entities — NOT loop over every item in a set. 

iteration The task should require the agent to perform an operation on EACH item in a set whose membership is discovered at runtime. The set is not known in advance — the agent must first query to find which items qualify, then process each one individually. Each iteration may involve multiple API calls. Examples: for each person who owes me money send a reminder, for every overdue task add a comment, for all notes tagged ‘urgent’ share them with a colleague, for each expense above $50 update its description. The task should NOT be solvable with a single bulk API call — individual per-item processing is required.

H.1 Per-Task, Per-App Histories 

For each task, the simulator maintains an independent rolling history of accepted (call, response) pairs for each app. Restricting the history to a single app sharply reduces prompt length and prevents irrelevant entities from leaking into one app’s simulated state. When constructing a prompt, the simulator formats the most recent calls in this app’s history as “Call 1

. . . Call N ,” each entry containing the API signature, the input parameters, and the previously simulated response. The history is capped at a configurable maximum number of calls, but in practice the per-task length stays well below this cap. When a task finishes, its histories are explicitly cleared, so simulator state never leaks between tasks. 

H.2 Deterministic Input Validation 

Inputs of an API call are validated against the API specification using rules that mirror the validation a real backend would apply. If any check fails, the simulator returns an error message without calling the LLM. The checks are: 

• Unknown parameters not present in the endpoint’s parameter list. 

• Missing required parameters for which no default was filled in. 

• Type mismatches , with the same scalar coercions a real backend would perform ( e.g. , the string “42” is accepted for an integer field, but a boolean is not; container types require an exact match). 

• Constraint violations , including length bounds, numeric bounds (with strict and non-strict inequalities), enumerated value sets, and email-format checks. Surfacing these errors deterministically rather than asking the LLM to invent them ensures that the agent always sees the same error message for the same malformed call, which makes downstream filtering and 28 learning more stable. 

H.3 Prompt Construction 

For every call that reaches the LLM, the simulator builds a two-message conversation: a fixed system prompt that lays out the simulator’s role, the six categories of context it receives, and the rules it must follow (Listing 7, with the rules block in Listings 5 and 6); and a per-call user prompt assembled from the template in Listing 8. The user prompt contains, in this order: the fully qualified API name, a JSON dump of the inputs (with defaults filled and the page-limit cap applied), the full API specification (description, parameters, response schema), the dedicated response-schema block extracted from the specification, which gives the model a clean target for its output, the task description that the teacher agent is currently solving, the per-app interaction history described in Sec. H.1; the user profile (first name, last name, email, phone number) drawn from the fictitious supervisor record assigned to this task, and the frozen current date and time. The model is then asked to return raw JSON. 

H.4 Output Schema Validation 

Once the LLM’s simulated response is received, we first strip any markdown code fences that the model may have emitted in violation of the rules, and then attempt a strict JSON parse. On parse failure, a structured feedback message containing the raw response and the parser error is appended to the conversation (Sec. H.6). After parsing, the JSON value is structurally compared against both (i) the documented success schema and (ii) the canonical error schema {"message" : string }. The response is accepted if it conforms to either, since both are legitimate outcomes according to the simulator rules (Listing 5). If neither schema matches, both errors are bundled into a structured feedback message, and the loop in Sec. H.6 retries. 

H.5 Simulator-judge 

A structurally valid response is still subject to a semantic check, but only for read-only (HTTP GET ) APIs. Write APIs ( POST , PATCH , DELETE ) are accepted as soon as they pass structural validation. This is because their semantic content is typically a simple acknowledgment whose validity is fully captured by the schema, and because gating writes behind a semantic judge would double the LLM cost for state-mutating calls without a proportionate gain in quality. For read APIs, the simulator invokes a separate LLM judge using the system prompt in Listing 9 and the user prompt in Listing 10. The judge receives the exact same context as the simulator (specification, inputs, task, history, user profile, and current date/time) alongside the candidate response, and is tasked with returning a JSON verdict containing three fields: a boolean validity flag, a list of issues, and a free-text suggestion block. The judge employs a bounded retry loop to handle any JSON-parsing failures of its own, after which it raises a simulation-failed error. 

H.6 Retry Loop With Structured Feedback 

The structural check and the judge feed into a single retry loop with two independent attempt budgets: a 

format budget for JSON-parse and schema-validation failures, and a semantic budget for judge rejections. After any rejection, the conversation is extended with two messages: an assistant message containing the rejected response verbatim, and a user message containing the structured feedback. For format failures the feedback names the precise error (parse exception, missing keys, type mismatch) and re-states the two acceptable shapes (success schema or error object). For semantic failures the feedback enumerates the judge’s listed issues and reproduces its suggestions verbatim. This in-context conversational format gives the model both a concrete negative example and an actionable hint. We find that it converges within the budget for the vast majority of calls. Once either budget is exhausted, we terminate trajectory synthesis.

## K Synthetic Application Suite 

To broaden the application coverage of AppWorld beyond its original nine productivity-focused apps, we construct a suite of 52 synthetic apps spanning ten functional categories. Each application is modeled as a REST API consistent with AppWorld’s structural conventions and is fully specified with typed endpoints, parameter constraints, and structured response schemas. The suite comprises 1,017 API endpoints across the 52 apps. 

K.1 Domain Selection 

The 52 app domains were selected through a manual curation process guided by two principles: real-world fidelity , where each domain corresponds to a widely used category of software, and functional diversity , where the collection as a whole requires agents to exercise qualitatively different interaction patterns, data models, and reasoning skills. Domains were selected to complement and extend AppWorld’s existing application coverage while minimizing redundancy with previously supported domains. We manually defined a seed set of 10 broad functional categories and identified specific app types within each category. For each app type we specified a short description and a list of key agent-facing functionalities. Listing 19 shows a representative entry from our seed set. Using these manually defined entries as in-context examples, we then prompted Gemini-2.5-Pro to expand the list with additional domains following the same format and coverage criteria, specifically requesting diversity across the remaining domain space and avoiding overlap with the seed entries. Table 22 summarizes the ten functional domains and their constituent app counts for the 52 synthetic apps.

Schemas that fail verification are discarded and regenerated up to a configurable retry limit.  

> Schema structure.

Each API is defined by eight fields: app_name , api_name , path , method (GET / POST /

PATCH / DELETE ), description , parameters , response_schemas , and canary_string . Specifications are authored using four Python helpers that enforce structural uniformity across all applications: api_() con-structs a full API entry; p() constructs a typed parameter; tok() is shorthand for the standard access_-token parameter; and paged() appends the three standard pagination parameters to a parameter list.  

> Design conventions.

We enforced the following conventions across all 52 applications to ensure a consistent interface vocabulary: 

• Authentication. Every application exposes a login API returning an access_token , which is required on all authenticated APIs, mirroring the pattern in the original AppWorld applications. 

• Pagination. All list-returning APIs accept page_index (0-based, default 0), page_limit (default 5), and sort_by (attribute prefixed with +/- for ascending/descending order). 

• Type-safe response schemas. Response schemas use typed placeholders: “string” for text fields, 1

for integer IDs, 0.0 for floats, “2019-01-01T00:00:00” for timestamps, and “user@example.com” for email fields, never domain-specific invented values. 

• Constraint notation. Parameter constraints follow a strict vocabulary: “length >= 1” , “value >= 0.0, <= 5.0” , “value in [’a’, ’b’]” , and “value is email address” . No other formats are valid. 

• Nested sub-resources. Where a resource logically contains an embedded sub-resource ( e.g. , a ride has a driver; a booking has a host), the response schema includes a nested object rather than flattening all fields. 

• Failure responses. The failure branch of every API is uniformly {“message”: “string”} across all 52 applications.

Gemini returns a JSON object mapping each api_name to its complete AppWorld-format entry. 32 Following conversion, each API entry is annotated with a canary string of the form appworld:{md5_-prefix}:{uuid} , where the MD5 prefix is the first seven hex characters of the application name’s hash and the UUID is freshly generated per API. The conversion pipeline includes automatic retry on JSON parse failures and falls back to a deterministic structural converter for any applications where LLM conversion fails, ensuring all 52 applications produce valid output.
